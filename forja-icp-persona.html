  • Scale-up com squads autônomos

  • TypeScript / Node.js (back + front)
  • React / Next.js (SPA/SSR)
  • Python (data, scripts, agentes)

  • AWS / GCP / Railway / Render
  • GitHub + CI/CD (Actions, Vercel, Fly.io)
  • Docker + Kubernetes (mid-stage)

📡 Canais de Descoberta

  • Twitter/X tech (dev influencers, demos virais)
  • Hacker News (Show HN + Ask HN)
  • YouTube (demos de fluxo real)
  • Product Hunt (lançamento)
  • Reddit r/webdev / r/MachineLearning
  • Comunidades Discord/Slack (Indie Hackers, etc.)
  • Indicação peer-to-peer (#1 canal B2B)

⚡ Comportamento de Compra

  • Trial-first obrigatório (sem cartão = preferido)
  • Decisão em 7–14 dias após primeiro uso real
  • ROI esperado em < 1 sprint (2 semanas)
  • Resistência a lock-in de dados
  • Expansão por adoção (PLG bottom-up)
  • Compra via cartão pessoal + reembolso

🆚 Concorrentes Diretos

  • Cursor/Copilot — pair programming (não AaaS)
  • Langchain/LangSmith — infra DIY complexa
  • CrewAI Cloud — orquestração sem memória nativa
  • AutoGen Studio — experimental, sem produção
  • Devin/SWE-agent — foco só em código
  • Diferenciar: memória cross-sessão + 15 agentes prontos + CLI local

04 — Debate Interno

War Room Notes

Debate real entre Gaia (estratégia), Hermes (pesquisa/dados) e Íris (produto/design) na construção deste documento.

🏛 War Room — Forja ICP Session

Olimpo · Gaia × Hermes × Íris · Jun 2026
DEBATE ATIVO
H
Hermes · Research Lead
Varri o mercado de developer tools SaaS 2025–2026. Insight crítico: 74% das decisões de compra de tools começam no dev individual (PLG), não no CTO. O modelo Forja de trial local-first é perfeitamente alinhado com esse comportamento. Mas atenção: o maior obstáculo não é preço — é tempo para perceber valor. LangChain perde usuários não por custo, mas porque o dev demora 2+ semanas para ver o primeiro resultado real. A Forja precisa garantir "primeiro valor em <30 minutos" como promessa explícita.
G
Gaia · Strategy
Concordo com o TTV (time-to-value), mas quero problematizar o ICP proposto. Estamos sendo conservadores demais no tamanho de time. Os dados do Hermes mostram que escale-ups de 120–200 devs também têm squads autônomos de 5–10 com budget próprio. O André Matsuda (Persona 3) representa isso — ele não espera aprovação do CFO. A Forja deveria ter um ICP primário (3–50 devs, decisão em dias) e um ICP secundário (squad autônomo dentro de empresa maior, decisão em semanas). Não são o mesmo motion de vendas.
Í
Íris · Product & Design
Do ponto de vista de produto, as 3 personas têm job-to-be-done distintos. Lucas quer reduzir tarefas repetitivas agora. Camila quer ter capacidade que ela não tem headcount para contratar. André quer desafogar o time de plataforma. Isso muda o onboarding, o primeiro caso de uso mostrado no trial e a mensagem do pricing. Para Lucas mostramos Hefesto (review de PR). Para Camila mostramos Atena (arquitetura). Para André mostramos Hermes (análise de logs/infra). O posicionamento "Stop building agents. Start using the best one." é forte — mas precisamos de 3 trilhas de onboarding diferenciadas no primeiro acesso.
H
Hermes · contra-ponto
Sobre a objeção da Camila ("Forja vai existir em 18 meses?"): isso é objeção de founder, não de usuário. Aparece mais em B2B SaaS pequeno, quase nunca em PLG de dev individual. A solução não é marketing — é ter GitHub público, roadmap visível e clientes referenciais nomeados. Um testimonial de CTO real vale mais que qualquer garantia contratual. Recomendo que o site Forja tenha uma seção "Quem usa" com logos reais antes de escalar aquisição.
G
Gaia · ponto estratégico
A objeção do André sobre InfoSec é a mais perigosa para o plano de go-to-market. Segurança de dados bloqueia escala para médias empresas. Minha recomendação estratégica: criar uma Forja Self-Hosted (Docker Compose, 1 comando) como opção no plano Team. Não precisa ser prioritária no lançamento, mas ter o roadmap comprometido publicamente remove o veto do InfoSec. Isso desbloqueia o ICP secundário (André) sem abrir mão da experiência cloud-first para o ICP primário (Lucas e Camila).
Í
Íris · design implication
Último ponto de produto: o canal de descoberta mostra que demos de fluxo real são o trigger principal. Não é landing page de texto. Lucas entrou pelo Twitter (demo de 2 min). André veio do YouTube (tutorial 20 min). Isso quer dizer que a Forja precisa de um canal de conteúdo técnico ativo — não marketing genérico. A Ayla gerando conteúdo sobre si mesma é o melhor demo possível. Sugiro: série "Ayla resolveu" no Twitter/LinkedIn mostrando casos reais de uso semanais. Custo marginal = zero. ROI = enorme.
✅ CONSENSO DO WAR ROOM
ICP primário: dev sênior/tech lead em startup 5–50 devs, stack TS/Python, já usa Copilot, sente dor de "contexto perdido". Ciclo de decisão de 7–14 dias, budget autoaprovável.

ICP secundário: squad autônomo de plataforma/infra dentro de empresa 50–200 devs — decisão em semanas, objeção de InfoSec requer roadmap self-hosted.

Prioridades de ação: (1) garantir "primeiro valor em <30 min" no onboarding; (2) 3 trilhas diferenciadas por persona no trial; (3) demos de fluxo real como principal canal de aquisição; (4) roadmap self-hosted publicado para desbloquear ICP secundário; (5) testimonials de CTOs reais antes de escalar mídia paga.
