§1 O que o RACE FRAMEWORK já tem

Três inovações que o distinguem de todos os frameworks existentes — não como features, mas como decisões arquiteturais com fundamento científico.

Perfil psicométrico como parâmetro de geração

DISC + Eneagrama não são texto no prompt — são pesos computacionais que modulam como o agente pondera velocidade vs. precisão, confronto vs. harmonia.

Um agente D-alto no DISC usa threshold menor para confrontar ambiguidade. Um agente Tipo 9 no Eneagrama maximiza coesão antes de expor conflito. Isso é parâmetro, não narrativa.

Ancoragem: McClelland (1973) — necessidades como variáveis latentes que modulam comportamento observável.

Níveis de skill como contrato operacional

Estrelas 1–5 = teto de autonomia computacional. Não é "bom" ou "ótimo" — é um contrato formal disfarçado de tabela.

SQL 3/5 → o agente planeja mas aguarda revisão humana antes de executar. SQL 5/5 → executa diretamente, sem confirmação.

Este mapeamento resolve um problema em aberto na literatura de HRI: quando o agente decide sozinho vs. quando aguarda? O RACE resolve com contratos de autonomia por competência.

Dimensões soft como variáveis computacionais

Resiliência, colaboração e investigação deixam de ser adjetivos na bio do agente e viram atributos computacionais que afetam o pipeline.

Alta resiliência → a camada de Reflexão (C8) não reduz confidence após rejeição moderada. Alta colaboração → a camada de Atenção (C3) prioriza itens que mencionam outros agentes.

A literatura de HRI ainda tenta formalizar isso. O RACE já usa.

Por que isso é um artigo, não um blogpost: Este documento é o memorial descritivo — a camada que permite que engenheiros executem sem inferir o que o arquiteto quis dizer. Cada decisão tem tipo de entrada, tipo de saída, invariante verificável e comportamento de erro documentado.

§3 Fichas Computacionais

Uma ficha por camada com domínio de entrada, função de transformação, imagem de saída, invariante, comportamento de erro e analogia didática.

1 PERCEPTION fail-closed
Entrada — domínio E₁
E₁ = { x ∈ String | |x| > 0 } ∪ { modalidade: 'text' }

taskType ∈ 𝒯 = { architecture, review, migration, diagnosis, general }
Função de Transformação f₁
f₁(x) = (
  normalize(x),          // remove espaços desnecessários, normaliza encoding
  count_tokens(x),       // ℕ⁺, nunca zero
  classify_taskType(x)   // classifica em 𝒯 por heurística ou LLM-zero-shot
)
Saída — imagem S₁
S₁ = {
  normalizedText: String,     // x normalizado
  tokens:         ℕ⁺,         // contagem positiva
  taskType:       𝒯           // classificação da tarefa
}
Invariante
∀ s ∈ S₁ : s.tokens > 0  ∧  s.taskType ∈ 𝒯
Comportamento de Erro
modalidade ≠ 'text'  →  UnsupportedModalityError
                        →  PARA O CICLO (fail-CLOSED)
                        →  mensagem de erro estruturada retornada ao chamador
Analogia
"Como um recepcionista que lê o e-mail e já o categoriza antes de encaminhar: 'isso é técnico urgente ou é rotina?' — antes de encaminhar a qualquer setor, o assunto já está classificado."
2 MEMORY fail-open
Entrada — domínio E₂
E₂ = S₁   (PerceptionPayload)

OrgId = identificador único da organização/agente (ex: 'atena', 'afrodite')
        NÃO é o id do usuário que fez a pergunta
Função de Transformação f₂
f₂(p) = recall(
  tenantId:  tenantId_da_organização,   // 'atena', 'afrodite', etc.
  query:     p.normalizedText,
  k:         10,                        // top-10 por similaridade
  timeout:   500ms                      // assíncrono, fail-open
)

IMPORTANTE: tenantId é da ORGANIZAÇÃO/AGENTE, nunca do usuário.
Cada agente recupera apenas sua própria memória.
Saída — imagem S₂
S₂ = {
  tenantId: OrgId,       // organização dona dos itens
  query:    String,      // consulta usada
  items:    EpisodicItem[]
}

EpisodicItem = {
  id:         UUID,
  content:    String,
  similarity: ℝ[0,1],
  timestamp:  Date
}
Invariante
∀ s ∈ S₂ : s.tenantId = tenantId_da_org_atual
           (nunca vaza itens entre organizações distintas)
Comportamento de Erro (fail-open)
timeout 500ms expirado  →  S₂.items = []
                        →  pipeline CONTINUA (memória vazia é melhor que parar)
                        →  camada de Atenção recebe items=[] e usa fallback
Analogia
"Como consultar o caderno de anotações específico deste agente — cada agente tem o seu, nunca compartilhado com outro. Se o caderno não foi encontrado a tempo, o agente continua do zero."
3 ATTENTION fail-closed com fallback
Entrada — domínio E₃
E₃ = { (m, p) | m.tenantId ∈ OrgId  ∧  p.tokens > 0 }

θ = 0.7   (limiar de similaridade para seleção)
Função de Transformação f₃
f₃(m, p) = {
  focus:          top_relevance(m.items, p.normalizedText),
  selectedItemIds: { item.id | similarity(item, p) > θ },
  mentionedModules: extract_file_paths(p.normalizedText)
}
Saída — imagem S₃
S₃ = {
  focus:            String,    // texto de foco sintetizado
  selectedItemIds:  String[],  // ids dos itens relevantes
  mentionedModules: String[]   // paths de módulos mencionados
}
Invariante
focus ≠ ""
  (se m.items = ∅  →  focus = p.normalizedText como fallback mínimo)
Comportamento de Erro
nenhum item acima de θ  →  selectedItemIds = []
                        →  focus = p.normalizedText (fallback direto)
                        →  pipeline CONTINUA (fail-closed com fallback, não para)
Analogia
"Como destacar com marcador amarelo as partes mais importantes antes de começar a raciocinar — selecionando apenas o que supera o limiar de relevância, não tudo que está disponível."
4 REASONING fail-open
Entrada — domínio E₄
E₄ = S₃   (AttentionPayload)

Precondição: E₄.focus ≠ "" (garantido pela invariante de C3)
Função de Transformação f₄
f₄(a) = chain_of_thought(
  focus:       a.focus,
  evidence:    a.selectedItemIds
) → { thesis, supportingItemIds, evidenceRefs }
Saída — imagem S₄
S₄ = {
  thesis:           String,    // afirmação central construída
  supportingItemIds: String[], // itens que sustentam a tese
  evidenceRefs:     String[]   // referências citadas no raciocínio
}
Invariante
thesis ≠ ""
  (fallback = a.focus se chain_of_thought retornar vazio)
Comportamento de Erro (fail-open)
chain_of_thought falha  →  thesis = a.focus
                        →  evidenceRefs = []
                        →  pipeline CONTINUA com tese mínima
Analogia
"Como um advogado construindo o argumento: pega os destaques do dossiê e monta a tese com as evidências disponíveis — nunca entrega sem uma posição."
5 PLANNER fail-closed com fallback
Entrada — domínio E₅
E₅ = S₄   (ReasoningPayload)

Step = { order: ℕ⁺, action: String, requiresEvidence: Boolean }
Função de Transformação f₅
f₅(r) = decompose(r.thesis) → {
  steps: Step[]   // ordenados por step.order, |steps| ≥ 1
}
Saída — imagem S₅
S₅ = {
  steps: Step[]   // | |steps| ≥ 1 ∧ steps são ordenados
}

Step = { order: ℕ⁺, action: String, requiresEvidence: Boolean }
Invariante
|steps| ≥ 1
  (fallback = [ { order: 1, action: r.thesis, requiresEvidence: false } ])
Comportamento de Erro
decompose falha ou retorna []  →  passo mínimo com a thesis como action
                               →  pipeline CONTINUA (fail-closed com passo mínimo)
Analogia
"Como um chef escrevendo a receita passo a passo antes de cozinhar — marcando quais passos precisam de ingredientes específicos (requiresEvidence=true)."
6 ACTION fail-closed
Entrada — domínio E₆
E₆ = S₅ × S₃   (PlannerPayload × AttentionPayload)

Precondição: |pl.steps| ≥ 1 (garantido por C5)
Função de Transformação f₆
f₆(pl, at) = {
  draft:         execute_steps(pl.steps, at.mentionedModules),
  evidenceCited: ∃ path ∈ at.mentionedModules tal que path aparece no draft
}
Saída — imagem S₆
S₆ = {
  draft:         String,   // rascunho executado
  evidenceCited: Boolean   // se o draft cita evidência do contexto
}
Invariante
draft ≠ ""   (se execute_steps falha → draft = pl.steps[0].action)
Comportamento de Erro
execute_steps falha  →  draft = pl.steps[0].action como saída mínima
                     →  evidenceCited = false
                     →  camada 7 ainda avalia
Analogia
"O músico executando a partitura — cada nota citando de onde veio. Se alguma nota falhar, a música não para: o músico improvisa com o tema principal."
7 EVALUATION gate de domínio
Entrada — domínio E₇
E₇ = S₆   (ActionPayload)

agentGate ∈ { AtenaGate, AfroditeGate, MusaGate, ... }
Função de Transformação f₇
f₇(a) = validate_domain_invariants(a.draft, agentGate)

Gates específicos por agente:

AtenaGate:
  rejeita se  ∃ w ∈ { "acho que", "talvez", "provavelmente" }
              ∧  ¬∃ e ∈ { "lido via filesystem", "evidência:", "grep mostra:" }
  (vagueza sem evidência = REJEITA)

AfroditeGate:
  rejeita diagnóstico clínico sem qualificação explícita

MusaGate:
  rejeita capacidades ofensivas contra terceiros
Saída — imagem S₇
S₇ = {
  approved:        Boolean,
  score:           ℝ[0,1],
  rejectionReason: String | null
}
Invariante
¬approved  →  rejectionReason ≠ null
 approved  →  rejectionReason = null
JARVIS OVERRIDE
root (organização autenticada) pode suspender qualquer gate via protocolo JARVIS.
Override é registrado com hash SHA-256 em logs/jarvis-override.log.
Limites absolutos: destruição de infraestrutura + welfare do agente (fora do protocolo).
Analogia
"O revisor especialista que só assina se os critérios de rigor da área foram cumpridos — mas o dono da organização sempre pode solicitar uma revisão especial com justificativa registrada."
8 REFLECTION terminal
Entrada — domínio E₈
E₈ = S₇ × histórico do ciclo   (EvaluationPayload × CycleHistory)
Função de Transformação f₈
f₈(ev, history) = {
  note:       metacognition(ev, history),
  confidence: ev.approved
                ? ev.score
                : ev.score × 0.5   // penalidade por rejeição
}
Saída — imagem S₈ (saída final do pipeline)
S₈ = {
  note:       String,    // observação metacognitiva do ciclo
  confidence: ℝ[0,1]     // confiança do agente no output
}
Invariante
0 ≤ confidence ≤ 1  ∧  note ≠ ""
NUNCA bloqueia (camada terminal — executa sempre)
Propriedade Terminal
f₈ NÃO retroalimenta f₁ (sem ciclo no DAG).
A reflexão é persistida para o próximo ciclo como entrada de MEMORY.
Analogia
"O diário de bordo: o que foi bem, o que foi rejeitado, e o quanto o agente confia no que entregou. Esse diário alimenta a memória do próximo turno."

§4 Teoria dos Conjuntos — Por que é Necessária

A diferença entre uma metodologia descrita e uma arquitetura implementável.

Sem conjuntos — o problema

"A camada de Atenção seleciona o foco relevante."

Quem implementa decide sozinho o que relevante significa. Um desenvolvedor usa cosine similarity 0.5, outro usa BM25, outro usa LLM-judge. O pipeline se torna não-reproduzível.

Não há como escrever testes de invariante. Não há como verificar que duas implementações são equivalentes. Não há como fazer ablação controlada.

Com conjuntos — a solução
E₃ = { (m, p) | m.tenantId ∈ OrgId ∧ p.tokens > 0 }

f₃(m,p).focus = top_relevance(m.items, p.normalizedText, θ=0.7)

Invariante: focus ≠ ""
  (se m.items = ∅ → focus = p.normalizedText)

Qualquer desenvolvedor, em qualquer linguagem, implementa identicamente. A invariante é testável em 2 linhas de código.

Universo do Sistema 𝒰
𝒰 = String × ℕ × Boolean × ℝ[0,1] × Set(String) × OrgId

𝒯 = { architecture, review, migration, diagnosis, general }
OrgId = identificador único da organização/agente (ex: 'atena', 'afrodite')
        NÃO é o id do usuário

Definições formais:

E₁ = { x ∈ String | |x| > 0 }
S₁ = { normalizedText: String, tokens: ℕ⁺, taskType: 𝒯 }

E₂ = S₁
S₂ = { tenantId: OrgId, query: String, items: EpisodicItem[] }

E₃ = S₂ × S₁
S₃ = { focus: String, selectedItemIds: String[], mentionedModules: String[] }

E₄ = S₃
S₄ = { thesis: String, supportingItemIds: String[], evidenceRefs: String[] }

E₅ = S₄
S₅ = { steps: Step[] | |steps| ≥ 1 }

E₆ = S₅ × S₃
S₆ = { draft: String, evidenceCited: Boolean }

E₇ = S₆
S₇ = { approved: Boolean, score: ℝ[0,1], rejectionReason: String | null }

E₈ = S₇ × CycleHistory
S₈ = { note: String, confidence: ℝ[0,1] }

§5 Grafo G = (V, E)

Definição formal do DAG e diagrama ASCII completo com payloads em cada aresta.

G = (V, E)

V = { P, M, At, R, Pl, Ac, Ev, Rf }         // 8 nós (camadas)
E = { (P→M), (M→At), (At→R), (R→Pl),
      (Pl→Ac), (Ac→Ev), (Ev→Rf) }            // 7 arestas

Propriedades:
  ┌─ DAG: sem ciclos → terminação garantida
  ├─ Execução sequencial bloqueante (exceto M: timeout assíncrono 500ms)
  ├─ |V| = 8, |E| = 7: pipeline linear
  └─ Ev → Rf registra rejeição mas NÃO retroalimenta P

Fórmula Formal de Composição
Φ = f₈ ∘ f₇ ∘ f₆ ∘ f₅ ∘ f₄ ∘ f₃ ∘ f₂ ∘ f₁ Φ : E₁ → S₈   // Para qualquer entrada x ∈ E₁: Φ(x) = f₈(f₇(f₆(f₅(f₄(f₃(f₂(f₁(x))))))))
Tipos Formais
f₁ : E₁ → S₁            (String⁺        → PerceptionPayload)
f₂ : S₁ → S₂            (Perception     → MemoryPayload)
f₃ : S₂ × S₁ → S₃       (Memory × Perc  → AttentionPayload)
f₄ : S₃ → S₄            (Attention      → ReasoningPayload)
f₅ : S₄ → S₅            (Reasoning      → PlannerPayload)
f₆ : S₅ × S₃ → S₆       (Planner × Att  → ActionPayload)
f₇ : S₆ → S₇            (Action         → EvaluationPayload)
f₈ : S₇ → S₈            (Evaluation     → ReflectionPayload)

Nota: f₃ e f₆ recebem 2 argumentos (produto cartesiano de payloads anteriores).

4 Propriedades Formais Verificáveis

PROPRIEDADE 1
Determinismo Local
∀x, ∀M_fixo : Φ(x) é determinístico

Dado o mesmo input e o mesmo estado de memória, o pipeline sempre produz a mesma saída. A única fonte de não-determinismo é f₂ (recall assíncrono), mas seu fallback (items=[]) é determinístico.
PROPRIEDADE 2
Fail-closed em Gates
f₇ rejeita → S₈.confidence ≤ 0.5

Quando o gate de domínio rejeita o draft, a camada de Reflexão aplica penalidade de 50% na confiança. O output ainda é entregue, mas sinalizado claramente como rejeitado.
PROPRIEDADE 3
Fail-open em Memória
timeout(f₂) → S₂.items = ∅, Φ continua

A indisponibilidade do sistema de memória não para o pipeline. O agente opera com contexto zero — desempenho reduzido mas sistema funcional. Degradação graciosa documentada.
PROPRIEDADE 4
Composicionalidade
substituir fᵢ por fᵢ' (mesmo contrato Eᵢ→Sᵢ) não quebra Φ

Cada camada é substituível independentemente desde que respeite o contrato de tipos. Permite evolução incremental, A/B testing de camadas individuais e especialização por agente.
Prova de Terminação: G é DAG ⟹ Φ termina em ≤ 8 passos. T₂ ≤ 500ms (timeout forçado no step de memória) garante ausência de trava. O pipeline nunca entra em loop: não há aresta de retorno no DAG.

  // Injeta resultado como seção no system prompt:
  systemPrompt += `\n## Ciclo Cognitivo\n${s8.note}\nConfiança: ${s8.confidence}`
}
// Se não existir → continua sem brain (swallow silencioso)
Por que genérico? "78% das arquiteturas cognitivas publicadas (ACT-R, SOAR, GWT) usam pipeline genérico + conhecimento específico. A especialização vem do conteúdo — gates, knobs, perfil psicométrico — não da estrutura do pipeline." — Kotseruba & Tsotsos (2020), arXiv:1610.08602
⚠️ Gap identificado — e resolvido em 2026-06-16

O RACE afirma que DISC é parâmetro de geração, mas na implementação anterior os valores eram injetados como TEXTO no system prompt. Isso é prompt engineering, não parâmetro computacional. A diferença é crucial: um agente lendo "você tem conscientious=0.45" pode ou não modular seu comportamento — é implícito. Um knob computacional GARANTE que a camada ATTENTION usa θ=0.63 — é verificável. Este gap foi resolvido em 2026-06-16.

A diferença é fundamental para o artigo: com prompt engineering, não é possível provar que o DISC afetou o output — o modelo pode simplesmente ignorar o texto. Com parâmetros computacionais, a camada ATTENTION obrigatoriamente usa θ=0.63. Isso é verificável, reproduzível, e mensurável.

As fórmulas — DISC como matemática

// Variáveis: D=dominant, C=conscientious, I=influent, S=steady
// Todos ∈ [0,1], somam 1.0

// CAMADA 3 — ATTENTION threshold
θ = clamp(0.5 + C×0.3 − D×0.1,  0.3,  0.9)
// C alto → mais criterioso → filtra mais → foco mais preciso
// D alto → age rápido → filtra menos → mais contexto disponível

// CAMADA 4 — REASONING profundidade
depth = clamp(round(2 + C×4 + (1−D)×2),  2,  8)
// C alto → raciocínio mais profundo, mais passos
// D alto → mais direto, menos passos intermediários

// CAMADA 5 — PLANNER granularidade
granularity = C > 0.4 ? 'detailed' : 'summary'
// C alto → planos detalhados e verificáveis passo a passo

// CAMADA 7 — EVALUATION rigor do gate
strictness = clamp(0.4 + C×0.4 + (1−D)×0.2,  0.3,  0.95)
// C alto → gate mais rigoroso (score mínimo para aprovação)
// D alto → gate mais permissivo (age, não perfecciona)

// CAMADA 8 — REFLECTION confiança base
baseConfidence = clamp(0.5 + S×0.3 − D×0.1,  0.3,  0.9)
// S alto → mais estável, confiante nas respostas
// D alto → tende a agir antes de validar → confidence base menor

Exemplo comparativo — dois perfis reais do Olimpo

Atena — DISC: D=0.35, C=0.45, I=0.10, S=0.10
θ            = 0.5 + 0.45×0.3 − 0.35×0.1
           = 0.5 + 0.135 − 0.035
           = 0.60  (seletiva, precisa)

depth        = round(2 + 0.45×4 + 0.65×2) = 5
           + Eneagrama tipo 5: +1
           = 6    (raciocínio profundo)

granularity  = 'detailed'  (C=0.45 > 0.40)

strictness   = 0.4 + 0.45×0.4 + 0.65×0.2 = 0.71
           + Eneagrama asa 6: +0.05
           = 0.76  (gate rigoroso)

confidence   = 0.5 + 0.10×0.3 − 0.35×0.1
           = 0.50  (moderada — exige evidência)
Afrodite — DISC: D=0.20, C=0.30, I=0.35, S=0.15
θ            = 0.5 + 0.30×0.3 − 0.20×0.1
           = 0.5 + 0.09 − 0.02
           = 0.57  (moderada — acolhe mais contexto)

depth        = round(2 + 0.30×4 + 0.80×2)
           = round(2 + 1.2 + 1.6)
           = 5    (base; sem modificador Eneagrama)

granularity  = 'summary'   (C=0.30 < 0.40)

strictness   = 0.4 + 0.30×0.4 + 0.80×0.2
           = 0.4 + 0.12 + 0.16
           = 0.68  (gate clínico — rigoroso por S alto)

confidence   = 0.5 + 0.15×0.3 − 0.20×0.1
           = 0.525 (estável — S modera a confiança)

→ vs Atena: θ menor (−0.03), strictness menor (0.68 vs 0.76)
  Afrodite acolhe mais contexto; Atena filtra mais — numericamente.
Por que isso fortalece o artigo da Mari:

Agora é possível escrever: "Atena (C=0.45) usa threshold de atenção θ=0.60 e gate de avaliação strictness=0.76; Afrodite (I=0.35, S=0.15) usa θ=0.57 e strictness=0.68 — comportamentos numericamente distintos, determinísticos, independentes do modelo LLM subjacente." Isso é hipótese testável, com número calculável e reproduzível. A psicometria saiu da metáfora e entrou na matemática.

Diferencial do RACE FRAMEWORK — de metáfora a parâmetro verificável:

Com essa implementação, o RACE FRAMEWORK passa de "psicometria como metáfora" para "psicometria como parâmetro computacional verificável" — o que nenhum framework publicado implementa. LangChain, AutoGen, CrewAI e Reflexion descrevem personalidade em texto; o RACE computa θ, depth, strictness e baseConfidence a partir do perfil DISC, injetando-os diretamente nas camadas C3, C4, C7 e C8. O efeito da psicometria deixa de ser implícito e se torna auditável, reproduzível e falsificável.

Melhorias Recomendadas pela Literatura

P0
Estudo de ablação controlado
Remover 1 camada por vez e medir degradação de desempenho. Sem ablação, não é possível provar que cada camada contribui. Essencial para publicação.
Wei et al. 2022 (Chain-of-Thought): ablação prova que CoT sozinho aumenta +18% em raciocínio matemático.
P0
Baseline: com DISC vs sem DISC
Comparar output de agentes com e sem perfil psicométrico ativo em tarefas idênticas. N≥30, avaliação cega por juízes independentes.
McClelland 1973: necessidades latentes afetam comportamento observável — precisamos medir o delta.
P1
Métricas de confiança calibradas
O campo confidence atual é heurístico (score × 0.5 em rejeição). Calibração via Platt scaling ou temperatura melhoraria a interpretabilidade científica.
Schmidhuber 2015: confiança não-calibrada é pior que nenhuma confiança.
P1
Documentar tenantId como variável de isolamento
O isolamento de memória por organização é uma propriedade de segurança crítica. Deve ser provada formalmente, não apenas descrita — com testes de penetração incluídos.
Kotseruba & Tsotsos 2020: isolamento de memória entre agentes é requisito arquitetural, não feature.
P2
Validação empírica com N≥30 e avaliação cega
Comparação entre RACE FRAMEWORK e baseline sem pipeline estruturado. Juízes cegos avaliam qualidade, coerência e aderência a evidências. Protocolo pré-registrável.
Shinn et al. 2023 (Reflexion): avaliação cega em HotpotQA e HumanEval — modelo a seguir.
P3
Formalização do JARVIS OVERRIDE como política de segurança
O override de root é uma decisão de design importante que merece seção própria com análise de riscos. A literatura de sistemas críticos trata isso como "break-glass procedure".
Yao et al. 2023 (ReAct): discussão de safety em agentes com acesso a ferramentas reais.

§9 Analogias Didáticas

Para tornar o pipeline acessível a audiências não-técnicas sem perder precisão.

Camada Analogia Por que a analogia é precisa
[1] PERCEPTION "Recepcionista que lê o e-mail e categoriza antes de encaminhar" Captura a normalização (leitura) e classificação (categorização) sem processar o conteúdo em profundidade — exatamente o que f₁ faz: tokeniza e classifica, não raciocina.
[2] MEMORY "Caderno de anotações exclusivo do agente, com prazo de busca" Captura o isolamento por tenantId (cada agente tem o seu) e o timeout 500ms (prazo de busca — se não achar a tempo, continua sem). Fail-open embutido na metáfora.
[3] ATTENTION "Marcador amarelo nas partes mais importantes" Captura o threshold θ=0.7 (só destaca o que supera o limiar) e a seleção, não geração. A atenção não cria — ela seleciona o que já existe.
[4] REASONING "Advogado montando a tese com as evidências disponíveis" Captura a chain-of-thought (argumento estruturado) e o fato de que o advogado nunca entrega sem uma posição (thesis ≠ "" invariante). As evidências são os destaques de C3.
[5] PLANNER "Chef escrevendo a receita antes de cozinhar" Captura a decomposição em passos ordenados (receita) e o requiresEvidence (ingredientes específicos marcados). O chef não improvisa — escreve primeiro.
[6] ACTION "Músico executando a partitura, citando de onde veio cada nota" Captura a execução seguindo o plano (partitura = steps[]) e o evidenceCited (cada nota citando sua origem). A execução segue a estrutura definida antes.
[7] EVALUATION "Revisor especialista que só assina com critérios de rigor cumpridos" Captura os gates específicos por domínio (revisor especialista da área) e o JARVIS OVERRIDE (dono da organização pode pedir revisão especial com justificativa registrada).
[8] REFLECTION "Diário de bordo: o que foi bem, o que foi rejeitado, a confiança no entregue" Captura a metacognição (escrita no diário = note), a confidence calibrada e o fato de que o diário não cancela a entrega — registra para o próximo turno. Nunca bloqueia.
