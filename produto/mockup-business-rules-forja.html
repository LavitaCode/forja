Gates & Sequência de Invoke

Ordem obrigatória de verificações a cada chamada ao agente

POST /api/v1/invoke_agent — sequência de gates
1.Auth (JWT válido?) → 401 se não autenticado
│
2.Plan check → effectivePlan(userId) ∈ {FREE, PRO, TEAM}
│
3.Agent whitelist → agentId ∈ allowedAgents[plan] → 403 se não
│
4.Usage check → count < maxPerWindow → 429 se excedido
│
5.Invoke agent (core/olimpo) ← lógica do agente aqui
│
6.incrementUsage() — FAIL-CLOSED: se falhar → NÃO retorna resposta
│
7.saveSessionTurn() — fail-open: retorna com ⚠️ context not saved se falhar
│
8.HTTP 200 + streaming output ao usuário
!
11ª invocação Free — Hard Block
Sem fallback para Qwen. Sem degradação. Free tier = acquisition tool, não produto degradado.
HTTP 429 · mensagem: "Renova dia {data}" · CTA Pro
!
Billing fail-closed
Se incrementUsage() falhar (Neon inacessível, race condition), a resposta do agente NÃO é entregue. Log BILLING_INCONSISTENCY. Usuário vê 500 "Billing error — try again".
Fail-closed · billing inconsistente = dívida técnica acumulada
⚠
Cartão recusado (Stripe)
Grace period 7 dias. 3 retries (D+0, D+3, D+5). Emails: D+0 (falha), D+3 (aviso), D+6 (downgrade iminente), D+7 (downgraded para Free).
D+7 sem pagamento → downgrade Free automático

Edge Cases & Regras Especiais

O que acontece em situações não-óbvias — documento para produto, QA e backend

O que é uma "invocação"?
Request iniciado pelo usuário e aceito pelo backend (POST aceito). Streaming conta como 1 invocação, independente de duração.
✓ Conta: request user-initiated aceita
Cancel mid-flight
Usuário cancela a request durante processamento (ctrl+C, fechar aba). A invocação JÁ foi iniciada — conta no quota.
✓ Conta: request já iniciada
Retry 5xx servidor
Retry automático por erro de infraestrutura (500, 502, 503) não consome quota. É falha do sistema, não do usuário.
✗ Não conta: falha do sistema
Retry manual pelo usuário
Usuário que re-tenta manualmente (nova request, não retry automático) é uma nova invocação e conta.
✓ Conta: nova request iniciada
Warm-up de memória timeout
Se carregar contexto do Neon demora >5s (timeout), invoke continua stateless sem contexto. NÃO conta como invocação se o agente não chegou a processar.
✗ Não conta: overhead interno
Sessão arquivada — restore
Se usuário invoca com --session de sessão arquivada (TTL expirado), restore silencioso automático + TTL reinicia. Não interrompe o fluxo.
✓ Restore automático silencioso
War Room — pool por invoke
Cada invoke no War Room = 1 do pool do tier, independente de quantos agentes respondem internamente. Orquestração interna não multiplica o custo.
✓ 1 invoke = 1 do pool (sempre)
Limites simultâneos
Se dois gates falham ao mesmo tempo (ex: quota + cartão recusado), a precedência é: Auth → Whitelist → Quota. Mensagem específica ao gate ativo.
✓ Precedência: auth → whitelist → quota

Compliance & LGPD

Direitos do usuário, retenção de dados e localização

Direito de Esquecimento
Usuário pode solicitar via dashboard ou suporte@forja.dev. Hard delete em 48h. Escopo: sessões, índice de projeto, perfil. Exceção: logs de billing (5 anos — obrigação fiscal BR).
Portabilidade de Dados
`forja session export --all --format json`. Inclui: session_id, agent_id, turns (prompt + response), timestamps. Não inclui: dados de billing, logs internos.
Auto-Archive & Purge
Email D-7 antes de auto-archive. Email D-7 antes de hard delete. Restauração sempre disponível até hard delete (self-serve).
Localização de Dados
Neon us-east-1 (atual). Migração sa-east-1 em Q3+. Disclosure no signup e na privacy policy.
